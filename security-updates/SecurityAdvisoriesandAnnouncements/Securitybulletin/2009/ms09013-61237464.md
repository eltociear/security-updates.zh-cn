Security Bulletin

Microsoft 安全公告 MS09-013 - 严重
==================================

Windows HTTP 服务中的漏洞可能允许远程执行代码 (960803)
------------------------------------------------------

发布时间: 2009年4月14日

**版本:** 1.0

### 一般信息

#### 摘要

此安全更新可解决 Microsoft Windows HTTP 服务 (WinHTTP) 中一个公开披露和两个秘密报告的漏洞。 此最严重的漏洞可能允许远程执行代码。 成功利用此漏洞的攻击者可以完全控制受影响的系统。 攻击者可随后安装程序；查看、更改或删除数据；或者创建拥有完全用户权限的新帐户。 那些帐户被配置为拥有较少系统用户权限的用户比具有管理用户权限的用户受到的影响要小。

对于 Microsoft Windows 2000、Windows XP、Windows Server 2003、Windows Vista 和 Windows Server 2008 的所有受支持版本，此安全更新的等级为“严重”。有关详细信息，请参阅本节的“**受影响和不受影响的软件**”小节。

此安全更新通过更改 Windows HTTP 服务处理错误和验证证书的方式，以及确保 Windows HTTP 服务可以正确使用 NTLM 凭据反射保护机制来消除漏洞。 有关漏洞的详细信息，请参阅下一节“**漏洞信息**”下面特定漏洞条目的“常见问题 (FAQ)”小节。

**建议。** 大多数客户均启用了“自动更新”，他们不必采取任何操作，因为此安全更新将自动下载并安装。 尚未启用“自动更新”的客户必须检查更新，并手动安装此更新。 有关自动更新中特定配置选项的信息，请参阅 [Microsoft 知识库文章 294781](http://support.microsoft.com/kb/294871)。

对于管理员、企业安装或者想要手动安装此安全更新的最终用户，Microsoft 建议客户使用更新管理软件立即应用此更新或者利用 [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) 服务检查更新。

另请参阅本公告后面部分中的“**检测和部署工具及指导**”一节。

**已知问题。** [Microsoft 知识库文章 960803](http://support.microsoft.com/kb/960803) 介绍了客户在安装此安全更新时可能遇到的当前已知问题。 本文还介绍了这些问题的建议解决办法。

此安全公告中描述的 Windows HTTP 服务凭据反射漏洞 [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550) 还影响 Internet Explorer 和 Windows Internet (WinINet) API。 Microsoft 建议 Internet Explorer 用户也安装[MS09-014](http://go.microsoft.com/fwlink/?linkid=146659)Internet Explorer 的累积性安全更新 (963027) 提供的安全更新。

**此更新为何消除了多个已报告的安全漏洞？**  
此更新之所以包含对某些漏洞的支持，是因为解决这些问题所需的修改位于相关的文件中。 客户只需安装此更新就可以了，而不用安装多个几乎相同的更新。

**此更新是否包含任何与安全性相关的功能更改？**  
是的，为解决 CVE-2009-0089 中描述的问题，Microsoft 在 SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp 下实施了一个新注册表项 UseFQDNFallbackForCnCompare，其可在就证书可分辨名称条目验证主机完全限定的域名时，控制 WinHTTP 的行为。

默认情况下，此注册表项不存在，这确保了 WinHTTP 可完全验证证书中包含的可分辨名称 (DN) 与启动该连接的 DN 相同。 原行为可通过将该注册表项设置为“True”重新启用。 在此情况下，验证将不那么严格，如果证书与 DNS 级别通过转发的名称更改匹配就会接受。 这是在安装此安全更新之前默认的不太安全的行为。

**我正在使用本安全公告中讨论的软件的较旧版本。 我该怎么办？**

Windows HTTP 服务整数下溢漏洞 - CVE 2009-0086  
---------------------------------------------
  
<span></span>  
Windows HTTP 服务处理远程 Web 服务器返回的特定值的方式中存在一个远程执行代码漏洞。 成功利用此漏洞的攻击者可以完全控制受影响的系统。 攻击者随后可安装程序；查看、更改或删除数据；或者创建权限与调用 WinHTTP API 以连接攻击者的 Web 服务器的服务或应用程序相同的新帐户。
  
要在“常见漏洞和披露”列表中以标准条目查看此漏洞，请参阅 [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086)。
  
#### Windows HTTP 服务整数下溢漏洞 (CVE-2009-0086) 的缓解因素
  
缓解是指一种设置、通用配置或者一般的最佳实践，它以默认状态存在，能够降低利用漏洞的严重性。 以下缓解因素在您遇到的情形中可能会有所帮助：
  
-   由于软件未使用 Windows HTTP 服务，Internet Explorer 的所有受支持版本不会受到此漏洞的影响。
  
#### Windows HTTP 服务整数下溢漏洞 (CVE-2009-0086) 的变通办法
  
Microsoft 并未发现此漏洞的任何变通办法。
  
#### Windows HTTP 服务整数下溢漏洞 (CVE-2009-0086) 的常见问题
  
**此漏洞的影响范围有多大？**    
这是一个远程执行代码漏洞。 成功利用此漏洞的攻击者可以完全控制受影响的系统。 攻击者随后可安装程序；查看、更改或删除数据；或者创建权限与调用 WinHTTP API 以连接攻击者的 Web 服务器的服务或应用程序相同的新帐户。
  
此漏洞影响 Windows 组件和使用 Windows HTTP 服务且配置为连接不受信任的远程 Web 服务器的第三方应用程序。
  
**造成漏洞的原因是什么？**    
漏洞是由未验证远程 Web 服务器返回的参数造成的。 这可能会导致任意代码执行。
  
**什么是 Windows HTTP 服务？**    
Windows HTTP 服务 (WinHTTP) 为开发人员提供 HTTP 客户端应用程序编程接口 (API)，以通过 HTTP 协议向 Web 服务器发送请求。 Microsoft Windows 组件和第三方软件均可使用 WinHTTP。 通用即插即用 (UPNP) 服务使用 WinHTTP。
  
**什么是通用即插即用？**    
通用即插即用 (UPnP) 是一种允许网络上的设备发现其他设备并确定如何与其配合的功能。 通过与大多数 Windows 用户很熟悉的即插即用 (PnP) 功能相比较，可以很容易了解 UPnP。 如果系统上安装了新硬件，PnP 允许操作系统检测它。 例如，当在计算机上安装新鼠标时，PnP 允许 Windows 检测它、加载所需的驱动程序和开始使用新鼠标。 UPnP 将此概念扩展到网络上的设备，而不是本地系统本身。 UPnP 让计算机学习网络上的其他设备，并确定如何使用他们。 例如，计算机可以使用 UPnP 检测网络上是否有它可以使用并知道如何使用的打印机。
  
**什么是简单服务发现协议 (SSDP)？**    
设备发现是一个过程，支持 UPnP 的计算机通过它可以知道他们可以使用的设备，并知道如何向他们请求服务。 当支持 UPnP 的计算机启动时，网络上可能已经存在它可以使用的设备。 要确定是否是这种情况，计算机将发出广播请求（称为 M-SEARCH 指令），请求子网内任何支持 UPnP 的设备直接响应它并提供有关使用它的信息。
  
同样，当支持 UPnP（例如，支持 UPnP 的打印机）的设备启动时，网络上可能已经存在想要使用它的支持 UPnP 的计算机。 该设备向子网内的所有计算机播出了一条消息（称为 NOTIFY 指令），宣布了它在网络上的存在并邀请计算机利用它提供的服务。
  
**攻击者可能利用此漏洞执行什么操作？**    
成功利用此漏洞的攻击者可以获得与调用 WinHTTP API 以连接攻击者的 Web 服务器的服务或应用程序相同的用户权限。 在被配置为拥有系统上较少用户权限的帐户下运行的应用程序，比在具有管理用户权限的帐户下运行的应用程序受到的影响要小。
  
**攻击者如何利用此漏洞？**    
攻击者会强迫或诱使使用 Windows HTTP 服务的程序连接到其恶意的 Web 服务器。 一定要注意通用即插即用 (UPnP) 服务使用的是 WinHTTP 库。 启用此服务后，本地子网上的恶意用户可能会响应 SSDP 请求，并将使用 WinHTTP 的 UPnP 服务连接到随后会利用此漏洞的恶意主机。 UPNP 服务在 LocalService 帐户下运行，该帐户是在系统上具有有限特权的帐户。
  
**受此漏洞威胁最大的系统有哪些？**    
工作站和终端服务器受到的威胁最大。 如果管理员允许用户登录服务器并运行程序，服务器可能面临更大风险。 不过，最佳做法是不要授予这样的权限。
  
**此更新有什么作用？**    
此更新通过修改 Windows HTTP 服务处理产生可利用条件的错误的方式来消除此漏洞。

否。 Microsoft 通过可靠的披露渠道了解到有关此漏洞的信息。
  
**发布此安全公告时，Microsoft 是否收到任何有关此漏洞已被利用的报告？**    
否。 在最初发布此安全公告时，Microsoft 未收到任何表明此漏洞已被公开用于攻击客户的信息，也没有看到任何发布的概念代码证明示例。

#### Windows HTTP 服务证书名称不匹配漏洞 (CVE-2009-0089) 的常见问题
  
**此漏洞的影响范围有多大？**    
这是在 Windows HTTP 服务中存在的一个欺骗漏洞。 当攻击者结合其他特定的攻击（如 DNS 欺骗）时，可能会成功欺骗使用 Windows HTTP 服务的任何应用程序所在的网站的数字证书。
  
**造成漏洞的原因是什么？**    
当一个应用程序调用 Windows HTTP 服务来设置远程 Web 服务器连接时，WinHTTP 将只针对主机尝试连接到的原始网站的证书验证 URL 的完全限定的域名。 与 DNS 欺骗的特定结合可能会将连接转到具有该特定主机（而不是 WinHTTP 启动该连接的网页）的有效证书的其他 Web 服务器。 DNS 欺骗与此漏洞的结合可能会导致 WinHTTP 将远程 Web 服务器的证书误认为可接受。 由于 WinHTTP 取决于用户交互的应用程序，该应用程序可能不会警告用户这种差异。
  
**什么是 Windows HTTP 服务？**    
Windows HTTP 服务 (WinHTTP) 为开发人员提供 HTTP 客户端应用程序编程接口 (API)，以通过 HTTP 协议向其他 HTTP 服务器发送请求。
  
**什么是数字证书？**    
数字证书是用于验证身份的常见凭据。 受信任的组织向证书的主体（个人或实体）分配证书。 颁发证书的受信任的组织称为证书颁发机构 (CA)，也称为证书颁发者。 可信赖的 CA 在验证证书的主体的身份之后才会颁发证书。
  
**什么是 DNS 欺骗？**    
域名系统 (DNS) 是一套包含 TCP/IP 的行业标准协议的其中之一。 DNS 是通过使用两个软件组件来实现的： DNS 服务器和 DNS 客户端（或者解析器）。 两个组件是作为背景服务应用程序运行。 网络资源是由数字 IP 地址标识，但是这些 IP 地址让网络用户很难记住。 DNS 数据库包含将网络资源的用户友好字母数字名称映射到这些资源用于通信的 IP 地址的记录。 因此，DNS 便充当一个助记键设备，使网络资源更容易被网络用户记住。 有关更多信息以及查看说明 DNS 如何适合其他 Windows 技术的逻辑图表，请阅读 TechNet 文章“[什么是 DNS？](http://technet.microsoft.com/en-us/library/cc787921.aspx)”
  
DNS 欺骗是一个广义攻击分类，在 DNS 系统中的某个位置，第三方引入不正确的 DNS 响应，但通常不负责这些响应。 这种攻击可能导致域或主机名解析为该名称所有者原本不打算解析的 IP 地址。 因此，主机可能被诱使连接到可能具有恶意的第三方服务器。
  
**攻击者可能利用此漏洞执行什么操作？**    
攻击者可能假冒安全的网站并向使用 Windows HTTP 服务的应用程序提供恶意内容，而让应用程序相信该内容是来自于安全的网站。
  
**攻击者如何利用此漏洞？**    
攻击者必须结合此漏洞与 DNS 欺骗或类似的攻击才能将出站连接引向他所控制的服务器。 然后他会提供一个对其网站有效的证书，如果用户连接该网站，则 Windows HTTP 服务将会对该用户最初尝试连接的目标主机网站误验证为有效。
  
**受此漏洞威胁最大的系统有哪些？**    
工作站和终端服务器受到的威胁最大。 如果管理员允许用户登录服务器并运行程序，服务器可能面临更大风险。 不过，最佳做法是不要授予这样的权限。
  
**此更新有什么作用？**    
此更新通过更改 Windows HTTP 服务验证属于安全网站的证书的方式来消除此漏洞

否。 Microsoft 通过可靠的披露渠道了解到有关此漏洞的信息。

Windows HTTP 服务凭据反射漏洞 - CVE-2009-0550  
---------------------------------------------
  
<span></span>  
当用户连接到攻击者的 Web 服务器时，Windows HTTP 服务处理 NTLM 凭据的方式中存在一个远程执行代码漏洞。 此漏洞允许攻击者重放用户凭据，并在登录用户的上下文中执行代码。 如果用户使用管理用户权限登录，成功利用此漏洞的攻击者便可完全控制受影响的系统。 攻击者可随后安装程序；查看、更改或删除数据；或者创建拥有完全用户权限的新帐户。 那些帐户被配置为拥有较少系统用户权限的用户比具有管理用户权限的用户受到的影响要小。
  
要在“常见漏洞和披露”列表中以标准条目查看此漏洞，请参阅 [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550)。
  
#### Windows HTTP 服务凭据反射漏洞 (CVE-2009-0550) 的缓解因素
  
缓解是指一种设置、通用配置或者一般的最佳实践，它以默认状态存在，能够降低利用漏洞的严重性。 以下缓解因素在您遇到的情形中可能会有所帮助：
  
-   在基于 Web 的攻击情形中，攻击者可能拥有一个网站，并在上面放置用来利用此漏洞的网页。 另外，接受或宿主用户提供的内容或广告的网站以及受到破坏的网站可能包含可能利用此漏洞的特制内容。 但是在所有情况下，攻击者无法强制用户访问这些网站。 相反，攻击者必须说服用户访问该网站，方法通常是让用户单击电子邮件或 Instant Messenger 消息中的链接以使用户链接到攻击者的网站。  
-   成功利用此漏洞的攻击者可以获得与本地用户相同的用户权限。 那些帐户被配置为拥有较少系统用户权限的用户比具有管理用户权限的用户受到的影响要小。  
-   Windows Vista 和 Windows Server 2008 的所有受支持版本将 Windows 防火墙的配置文件默认设置为公开，这就减少了在这些平台上受此漏洞影响的风险，因为在公共配置文件下，防火墙阻止了 SMB 通信。 由于客户端必须运行接受 NTLM 身份验证的服务，才能使攻击成功、反映凭据，最常见的攻击媒介可能会是 SMB。
  
#### Windows HTTP 服务凭据反射漏洞 (CVE-2009-0550) 的变通办法
  
Microsoft 并未发现此漏洞的任何变通办法。
  
#### Windows HTTP 服务凭据反射漏洞 (CVE-2009-0550) 的常见问题
  
**此漏洞的影响范围有多大？**    
这是一个远程执行代码漏洞。 成功利用此漏洞的攻击者可以完全控制受影响的系统。 攻击者可随后安装程序；查看、更改或删除数据；或者创建拥有完全用户权限的新帐户。

**验证是否已应用更新**

-   **Microsoft Baseline Security Analyzer**

    要验证受影响的系统上是否应用了安全更新，可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具。 有关详细信息，请参阅本公告前面的“**检测和部署工具及指导**”一节。

-   **文件版本验证**

    由于 Microsoft Windows 有多个版本，因此以下步骤可能会与您的系统上的步骤不同。 如果不同，请参阅产品文档来完成这些步骤。

    1.  单击“**开始**”，然后单击“**搜索**”。
    2.  在“**搜索结果**”窗格中，单击“**搜索助理**”下的“**所有文件和文件夹**”。
    3.  在“**完整或部分文件名**”框中，键入相应文件信息表中的一个文件名，然后单击“**搜索**”。
    4.  在文件列表中，用鼠标右键单击相应文件信息表中的一个文件名，然后单击“**属性**”。
        **注意** 根据系统上安装的操作系统或程序的版本，文件信息表中所列的某些文件可能并未安装。
    5.  在“**版本**”选项卡上，通过将系统上安装的文件与相应文件信息表中记录的版本进行比较来确定该文件的版本。
        **注意** 在安装过程中，除文件版本以外的其他属性均可能会发生变化。 不支持通过将其他文件属性与文件信息表中的信息进行比较这种方法来验证是否应用了更新。 另外，在某些情况下，在安装过程中还可能会重命名文件。 如果没有文件或版本信息，请使用另外一种方法来验证更新安装。

-   **注册表项验证**

    您也可以通过查看本节“**参考表**”中列出的注册表项来验证此安全更新安装的文件。

    这些注册表项可能不包含已安装文件的完整列表。 另外，如果管理员或 OEM 将此安全更新集成或组合到 Windows 安装源文件中，则可能无法正确创建这些注册表项。

**验证是否已应用更新**

-   **Microsoft Baseline Security Analyzer**

    要验证受影响的系统上是否应用了安全更新，可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具。 有关详细信息，请参阅本公告前面的“**检测和部署工具及指导**”一节。

-   **文件版本验证**

    由于 Microsoft Windows 有多个版本，因此以下步骤可能会与您的系统上的步骤不同。 如果不同，请参阅产品文档来完成这些步骤。

    1.  单击“**开始**”，然后单击“**搜索**”。
    2.  在“**搜索结果**”窗格中，单击“**搜索助理**”下的“**所有文件和文件夹**”。
    3.  在“**完整或部分文件名**”框中，键入相应文件信息表中的一个文件名，然后单击“**搜索**”。
    4.  在文件列表中，用鼠标右键单击相应文件信息表中的一个文件名，然后单击“**属性**”。
        **注意** 根据系统上安装的操作系统或程序的版本，文件信息表中所列的某些文件可能并未安装。
    5.  在“**版本**”选项卡上，通过将系统上安装的文件与相应文件信息表中记录的版本进行比较来确定该文件的版本。
        **注意** 在安装过程中，除文件版本以外的其他属性均可能会发生变化。 不支持通过将其他文件属性与文件信息表中的信息进行比较这种方法来验证是否应用了更新。 另外，在某些情况下，在安装过程中还可能会重命名文件。 如果没有文件或版本信息，请使用另外一种方法来验证更新安装。

-   **注册表项验证**

    您也可以通过查看本节“**参考表**”中列出的注册表项来验证此安全更新安装的文件。

    这些注册表项可能不包含已安装文件的完整列表。 另外，如果管理员或 OEM 将此安全更新集成或组合到 Windows 安装源文件中，则可能无法正确创建这些注册表项。

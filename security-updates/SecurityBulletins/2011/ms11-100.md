Microsoft 安全公告 MS11-100 - 严重
==================================

.NET Framework 中的漏洞可能允许特权提升 (2638420)
-------------------------------------------------

发布时间: 2011年12月29日 | 更新时间: 2012年7月10日

**版本:** 1.6

### 一般信息

#### 摘要

此安全更新可解决 Microsoft .NET Framework 中一个公开披露的漏洞和三个秘密报告的漏洞。如果未通过身份验证的攻击者传送特制 Web 请求至目标站点，最严重的漏洞可能允许特权提升。成功利用此漏洞的攻击者可以在 ASP.NET 站点上在一个现有帐户的上下文内容中执行任何操作，包括执行任意命令。要利用此漏洞，攻击者必须能在 ASP.NET 站点上注册一个帐户，且必须知道一个现有用户名。

对于 Microsoft Windows 所有受支持版本上的 Microsoft .NET Framework 1.1 Service Pack 1，Microsoft .NET Framework 2.0 Service Pack 2，Microsoft .NET Framework 3.5 Service Pack 1，Microsoft .NET Framework 3.5.1 和 Microsoft .NET Framework 4，此安全更新的等级为“严重”。有关详细信息，请参阅本节中“**受影响和不受影响的软件**”小节。

该安全更新通过更正 .NET Framework 处理特制请求的方式和 ASP.NET Framework 验证用户身份和处理缓存内容的方式来解决该漏洞。有关这些漏洞的详细信息，请参阅下一节“**漏洞信息**”下面特定漏洞条目的“常见问题 (FAQ)”小节。

此安全更新也解决了最初在 [Microsoft 安全通报 2659883](http://technet.microsoft.com/security/advisory/2659883) 中描述的漏洞。

**建议。** 大多数客户均启用了“自动更新”，他们不必采取任何操作，因为此安全更新将自动下载并安装。尚未启用“自动更新”的客户必须检查更新，并手动安装此更新。有关自动更新中特定配置选项的信息，请参阅 [Microsoft 知识库文章 294871](http://support.microsoft.com/kb/294871)。

对于管理员、企业安装或者想要手动安装此安全更新的最终用户，Microsoft 建议客户考虑使用更新管理软件应用此更新或者使用 [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) 服务检查更新。

另请参阅本公告后面部分中的“**检测和部署工具及指导**”一节。

**已知问题。**[Microsoft 知识库文章 2638420](http://support.microsoft.com/kb/2638420) 介绍了客户在安装此安全更新时可能遇到的当前已知问题。本文还介绍了这些问题的建议解决办法。

#### 受影响和不受影响的软件

已对下列软件进行测试，以确定受到影响的版本。其他版本的支持生命周期已结束或者不受影响。要确定软件版本的技术支持生命周期，请访问 [Microsoft 技术支持生命周期](http://go.microsoft.com/fwlink/?linkid=21742)。

与此安全更新相关的常见问题 (FAQ)
--------------------------------

**为什么在** **2012 年** **5 月** **22 日修订此公告？**  
Microsoft 修订此公告是为了宣布针对适用于 Microsoft .NET Framework 2.0 Service Pack 2 的 KB2656352 提供检测更改以更正安装问题。这只是检测更改。没有更改安全更新文件。已经成功更新了其系统的客户不需要执行任何操作。

**此公告是否与 MS12-035 .NET Framework 中的漏洞可能允许远程执行代码有关　(2693777)？**  
是。此公告中所提供的　KB2656353　也可以解决记录在　[MS12-035](http://go.microsoft.com/fwlink/?linkid=246970)　中的 CVE-2012-0160 和 CVE-2012-0161。

**Windows Developer Preview 上的** **Microsoft .NET Framework 3.5 和 Microsoft .NET Framework 4.5** **是否受此漏洞** **影响？**  
是。此漏洞是在 Windows Developer Preview　发布之后报告的。鼓励运行 Developer Preview 软件的客户下载并向其系统应用此更新。有关详细信息，包括 Microsoft 下载中心链接，请参阅以下 Microsoft 知识库文章：

| 受影响的软件                                                | Microsoft 知识库文章                                                    |
|-------------------------------------------------------------|-------------------------------------------------------------------------|
| Windows Developer Preview 上的 Microsoft .NET Framework 3.5 | [Microsoft 知识库文章 2656360](http://support.microsoft.com/kb/2656360) |
| Windows Developer Preview 上的 Microsoft .NET Framework 4.5 | [Microsoft 知识库文章 2656354](http://support.microsoft.com/kb/2656354) |

安全更新可从 [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) 和 [](http://go.microsoft.com/fwlink/?linkid=21130)[Windows Update](http://go.microsoft.com/fwlink/?linkid=21130)</a> 获得。

**此更新是否包含任何与安全性相关的功能更改？**  
是。除了本公告“漏洞信息”部分列出的更改外，此更新还包含 ASP.NET 框架的功能更改。此更新将表单项、文件和 JSON 成员的最大数量请求限制为 1000。此更改会导致 ASP.NET 应用程序拒绝具有超过 1000 个此类元素的请求。请求这些类型的客户端将被拒绝，并且浏览器中会出现一条错误消息（通常为一条错误 500 的消息）。此限制可以根据应用程序进行配置。请参阅 [Microsoft 知识库文章 2661403](http://support.microsoft.com/kb/2661403)。

此更新还能以与较旧版本的表单身份验证票证不兼容的方式更改表单身份验证票证的格式。例如，安装此更新后，在有些服务器更新而有些不会更新的 Web 场环境中，更新的服务器可能会生成在未更新的服务器上不兼容的表单身份验证票证。请参阅 [Microsoft 知识库文章 2661404](http://support.microsoft.com/kb/2661404)。

**在哪里可以找到文件信息详细资料？**  
有关文件信息详细资料的位置，请参阅“**安全更新部署**”部分中的参考表。

**我如何确定安装了哪个版本的 Microsoft .NET Framework？**  
您可以在一个系统上安装和运行多个版本的 .NET Framework，也可以按任何顺序安装各个版本。有多种方式确定当前安装的 .NET Framework 版本。有关详细信息，请参阅 [Microsoft 知识库文章 318785](http://support.microsoft.com/kb/318785)。

**.NET Framework 4 和 .NET Framework 4 客户端配置文件有什么区别？**  
两种配置文件中提供 .NET Framework 版本 4 可再次分发程序包： .NET Framework 4 和 .NET Framework 4 客户端配置文件。.NET Framework 4 客户端配置文件是针对客户端应用程序进行优化的 .NET Framework 4 配置文件的子集。它提供适用于大多数客户端应用程序的功能，包括 Windows Presentation Foundation (WPF)、Windows Forms、Windows Communication Foundation (WCF) 和 ClickOnce 功能。这可加快针对 .NET Framework 4 客户端配置文件的应用程序的部署速度并减小安装程序包。有关详细信息，请参阅 MSDN 文章“[.NET Framework 客户端配置文件](http://msdn.microsoft.com/en-us/library/cc656912.aspx)”。

**我的系统上安装的 Microsoft .NET Framework 的版本列出了两个更新。我需要两个更新都安装吗？**  
是。客户应该应用为他们的系统上安装的软件提供的所有更新。

**我需要按特定顺序安装这些安全更新吗？**  
否。.NET Framework 的一个版本的多个更新可以按任意顺序应用。我们建议 .NET Framework 的不同版本的多个更新按从最低版本号到最高版本号的顺序应用，但是并不要求按照该顺序进行。

**我安装了 .NET Framework 3.0 Service Pack 2；此公告中的受影响软件中没有列出此版本。我是否需要安装更新？**  
此公告描述了 .NET Framework 2.0 功能层和 .NET Framework 4 中的一个漏洞。.NET Framework 3.0 Service Pack 2 安装程序链接在 .NET Framework 2.0 Service Pack 2 安装程序中，因此安装前者也就安装了后者。因此，安装了 .NET Framework 3.0 Service Pack 2 的客户需要安装 .NET Framework 2.0 Service Pack 2 的安全更新。

**我安装了 .NET Framework 3.5 Service Pack 1。我需要安装任何其他更新吗？**  
此公告描述了 .NET Framework 2.0 功能层和 .NET Framework 4 中的一个漏洞。.NET Framework 3.5 Service Pack 1 安装程序链接在 .NET Framework 2.0 Service Pack 2 安装程序和 .NET Framework 3.0 Service Pack 2 安装程序中。因此，安装了 .NET Framework 3.5 Service Pack 1 的客户还需要安装 .NET Framework 2.0 Service Pack 2 的安全更新。

**我正在使用本安全公告中讨论的软件的较旧版本。我该怎么办？**  
已对本公告中列出的受影响的软件进行测试，以确定受到影响的版本。其他版本的支持生命周期已结束。有关产品生命周期的详细信息，请访问 [Microsoft 产品技术支持生命周期](http://go.microsoft.com/fwlink/?linkid=21742)网站。

使用该软件的较旧版本的客户应优先考虑迁移到受支持的版本，以防止可能会受到新出现漏洞的影响。要确定您的软件版本的技术支持生命周期，请参阅[选择一项产品以获取生命周期信息](http://go.microsoft.com/fwlink/?linkid=169555)。有关这些软件版本的 Service Pack 的详细信息，请参阅[Service Pack 生命周期支持策略](http://go.microsoft.com/fwlink/?linkid=89213)。

如果用户需要获得较旧软件的定制支持，则必须与其 Microsoft 客服小组代表、其技术客户经理或适当的 Microsoft 合作伙伴代表联系以了解定制支持选项。没有联合合同、优先支持合同或授权合同的客户可与其当地的 Microsoft 销售分支机构联系。有关联系信息，请访问 [Microsoft Worldwide Information](http://go.microsoft.com/fwlink/?linkid=33329) 网站，在联系信息列表中选择国家/地区，然后单击“**Go**”以查看电话号码列表。在拨打电话时，请找当地的“企业技术咨询支持服务”销售经理进行洽谈。有关详细信息，请参阅 [Microsoft 技术支持生命周期策略常见问题](http://go.microsoft.com/fwlink/?linkid=169557)。

HashTable 中的冲突可能导致 DoS 漏洞 - CVE-2011-3414
---------------------------------------------------

ASP.NET Framework 处理特制请求的方式中存在一个拒绝服务漏洞，从而导致哈希冲突。成功利用此漏洞的攻击者可以向 ASP.NET 服务器发送少量特制请求，致使性能大幅降低到足以导致拒绝服务条件。

要在“常见漏洞和披露”列表中以标准条目查看此漏洞，请参阅 [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414)。

#### HashTable 中的冲突可能导致 DoS 漏洞 (CVE-2011-3414) 的缓解因素

缓解是指一种设置、通用配置或者一般的最佳实践，它以默认状态存在，能够降低利用漏洞的严重性。以下缓解因素在您遇到的情形中可能会有所帮助：

-   在任何 Windows 操作系统上，IIS 默认为未启用
-   禁止 application/x-www-form-urlencoded 或 multipart/form-data HTTP 内容类型的站点不易受到攻击
-   默认情况下，安装 .NET Framework 而不安装 ASP.NET。只有手动安装和启用 ASP.NET 的客户才可能受此问题的影响。

#### HashTable 中的冲突可能导致 DoS 漏洞 (CVE-2011-3414) 的变通办法

变通办法是指一种设置或配置更改，它不能从根本上纠正漏洞，但有助于在应用更新之前封堵已知的攻击源。Microsoft 已测试了以下变通方法，并在讨论中指明了变通方法是否会降低功能性：

-   **基于配置的变通办法**

    以下变通办法配置 ASP.NET 将从客户端接受的最大请求大小的限制。降低最大请求大小将降低 ASP.NET 服务器遭受拒绝服务攻击的易受攻击程度。

    **注意：** 请在对其作出任何更改之前备份您的配置。

    此配置值可以通过将条目添加到根 web.config 或 applicationhost.config 而全局应用到所有 ASP.NET 站点。此外，还可以将此配置添加到特定站点或应用程序的 web.config 文件而限定于该站点或该应用程序。有关 ASP.NET 配置的详细信息，请参考 MSDN 文章[ASP.NET 配置概述](http://msdn.microsoft.com/en-us/library/ms178683.aspx)。

    1.  如果您的应用程序使用 ViewState，请在相应的 ASP.NET 配置文件中添加以下配置，以限制 ASP.NET 将从客户端接受的最大请求大小为 200 KB。

        ```
        <configuration>
        <system.web>
        <httpRuntime maxRequestLength="200”/>
        </system.web>
        </configuration>
        
        ```
    2.  如果您的应用程序未使用ASP.NET ViewState，请在相应的 ASP.NET 配置文件中添加以下配置，以限制 ASP.NET 将从客户端接受的最大请求大小为 20 KB。

        ```
        <configuration>
        <system.web>
        <httpRuntime maxRequestLength="20”/>
        </system.web>
        </configuration>

        ```
    有关 maxRequestLength 配置选项的详细信息，请参阅 [httpRuntime Element](http://msdn.microsoft.com/en-us/library/e1f13641.aspx) 上的 MSDN 文档。

    **注意：** 增加默认的 HTTP 查询字符串和请求标头大小限制，将使服务器更容易受此通报中描述的拒绝服务问题的影响。有关如何设置这些限制的详细信息，请参阅 IIS 的[Http.sys 注册表设置。](http://support.microsoft.com/kb/820129)

    **变通办法的影响：** 超过配置的 maxRequestLength 的客户端请求将导致服务器端出现一则 ConfigurationErrorsException 消息和一个返回到客户端的 http 错误状态。以上配置的限制将用作一个基准并且可以进行调整以避免应用程序失败。以一个更小值配置 maxRequestLength 值将降低服务器受此特定拒绝服务影响的程度 可能受此变通办法影响的应用程序是那些接受客户端文件上载以及生成大视图状态的应用程序。

    应用此变通办法后，在应用程序事件日志中可能会出现下面的客户端请求失败。

    ```
    事件代码： 3004 
    事件消息： 表单大小超过允许的限制。 
    异常信息： 
    异常类型： HttpException 
    异常消息： 超过最大的请求长度。
    at System.Web.HttpRequest.GetEntireRawContent()
    at System.Web.HttpRequest.FillInFormCollection()
    at System.Web.HttpRequest.get_Form()
    at System.Web.HttpRequest.get_HasForm()
    at System.Web.UI.Page.GetCollectionBasedOnMethod(Boolean dontReturnNull)
    at System.Web.UI.Page.DeterminePostBackMode()
    at System.Web.UI.Page.ProcessRequestMain(Boolean includeStagesBeforeAsyncPoint, Boolean includeStagesAfterAsyncPoint)
    ```

    此外，请求失败会导致 IIS 日志文件中出现下面的条目。

    ```
    #Fields:s-ip cs-method cs-uri-stem cs-uri-query s-port cs-username c-ip cs(User-Agent) sc-status sc-substatus sc-win32-status time-taken 

    127.0.0.1 POST /webSite/Default.aspx - 80 - 127.0.0.1 - 500 0 0 1268
    ```

    **删除变通办法：** 要撤消变通方法，请删除此变通办法中添加的配置部分或恢复为变通办法之前的配置备份。

#### HashTable 中的冲突可能导致 DoS 漏洞 (CVE-2011-3414) 的常见问题

**此漏洞的影响范围有多大？**  
这是一个拒绝服务漏洞。此漏洞允许未经身份验证的攻击者降低创建拒绝服务条件的 ASP.NET 站点的性能。

**造成此漏洞的原因是什么？**  
由于 ASP.NET 哈希特制请求并将该数据插入导致哈希冲突的哈希表的方式而导致存在此漏洞。当很多这些冲突相互链接在一起时，哈希表的性能会大幅降低而导致拒绝服务条件。

**攻击者可能利用此漏洞执行什么操作？**  
攻击者可能利用此漏洞引发拒绝服务攻击，破坏使用 ASP.NET 的站点的可用性。

**攻击者如何利用此漏洞？**  
未经身份验证的攻击者可以向受影响的 ASP.NET 站点发送少量特制 ASP.NET 请求，导致拒绝服务条件。

**受此漏洞威胁最大的系统有哪些？**  
安装了 ASP.NET 的面向 Internet 的系统受此漏洞的威胁最大。此外，使用 ASP.NET 表单身份验证的内部网站也会受到此漏洞的影响。

**此更新有什么作用？**  
此更新通过更正 .NET Framework 处理特制请求的方式来解决此漏洞。

**发布此安全公告时，此漏洞是否已公开披露？**  
是。此漏洞已公开披露。已为此漏洞分配了常见漏洞和披露号码 [CVE-2011-3414](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3414)。

**发布此安全公告时，Microsoft 是否收到任何有关此漏洞已被利用的报告？**  
否。在最初发布此安全公告时，Microsoft 未收到任何表明此漏洞已公开用于攻击用户的信息。

**这是**一个行业问题？
针对此类漏洞的攻击一般称为哈希冲突攻击。此类攻击并不专门针对于 Microsoft 技术和影响其他 Web 服务软件供应商。客户应联系其 Web 平台供应商以获得指导或更新。

-   **在 web.config 中禁用 Forms Authentication**

    2.  将身份验证模式设置为：Windows、Passport 或无。

        ```
        <system.web>

        <身份验证模式 =“Windows | Passport | 无/>”

        </system.web>
        ```

        关于每个模式的详细信息，请访问[http://msdn.microsoft.com/en-us/library/aa291347(v=vs.71).aspx。](http://msdn.microsoft.com/en-us/library/aa291347(v=vs.71).aspx)

        **如何撤消变通方法。**

        从备份副本恢复 web.config 文件。

#### ASP .Net 身份验证绕过漏洞 (CVE-2011-3416) 的常见问题

**此漏洞的影响范围有多大？**  
这是一个特权提升漏洞。此漏洞中，未经身份验证的远程攻击者可以进入 ASP.NET 应用程序上的另一个帐户，使攻击者能在目标用户的环境中的站点上执行任意命令。

**造成此漏洞的原因是什么？**  
ASP.NET Framework 验证特制用户名时出错，会导致此漏洞产生。

**攻击者可能利用此漏洞执行什么操作？**  
成功利用此漏洞的攻击者可以在目标用户的环境中采取任何行动，包括执行任意命令。

**攻击者如何利用此漏洞？**  
若要利用这个漏洞，未经身份验证的攻击者首先需要知道一个现有帐号名称，以便在站点上进行模拟。然后攻击者可以使用之前注册的账户名称制作特殊的 Web 请求，进入那个账户。然后攻击者可以在目标用户的上下文中采取任何行动，包括在站点上执行任意命令。

**受此漏洞威胁最大的系统有哪些？**  
安装了 ASP.NET 的面向 Internet 的系统受此漏洞的威胁最大。此外，使用 ASP.NET 表单身份验证的内部网站也会受到此漏洞的影响。

**如果我的网站使用一个 Web 群，我需要更新 Web 群中的所有** **计算机吗？**  
是。此更新更改了表单身份验证票证的格式，新格式与以前格式不兼容。这意味着使用旧行为的计算机无法读取使用新行为生成的票证，反之亦然。应用程序使用基于表单的身份验证的管理员在部署此安全更新时，必须执行特定步骤以确保所有服务器同时切换至新行为。

有关详细信息，请参阅 [Microsoft TechNet 文章 2659968](http://support.microsoft.com/kb/2659968)。

**更新 web 场中的** **计算机时，是否能够每次更新一台** **而不是同时** **更新所有计算机？**  
是。如果无法同时将 MS11-100 部署到指定的 Web 场中的所有服务器，您可以在安装更新之前在 web.config 或 machine.config 文件中设置一个兼容性开关，从而在安装更新之后强制执行旧行为。要使用兼容性开关如下：

```
<appSettings> 
<add key="aspnet:UseLegacyFormsAuthenticationTicketCompatibility" value="true" /> 
</appSettings>
```

设置此开关将允许仅更新了部分计算机的 Web 场继续正常运行。但是请注意，设置了配置开关的计算机将处于不安全状态，并且将不能通过此安全更新中的修补程序解决。在将 MS11-100 部署到场中的所有计算机后，必须删除配置开关以启用新的安全行为。

**此更新是否支持我的站点的表单身份验证 ticketCompatibilityMode 设置？**  
否。由于此安全更新中的修补程序更改表单身份验证票证的格式，安装 MS11-100 之后就不再支持 ticketCompatibilityMode 配置设置。

**此更新有什么作用？**  
此更新通过更正 ASP.NET Framework 验证用户的方式解决此漏洞。

**发布此安全公告时，此漏洞是否已公开披露？**  
否。Microsoft 通过协调漏洞披露渠道了解到有关此漏洞的信息。

**发布此安全公告时，Microsoft 是否收到任何有关此漏洞已被利用的报告？**  
否。在最初发布此安全公告时，Microsoft 未收到任何表明此漏洞已公开用于攻击用户的信息。

ASP.NET 表单身份验证票证缓存漏洞 - CVE-2011-3417
------------------------------------------------

当使用表单验证和可调用期时，ASP.NET Framework 处理缓存内容的方式中存在特权提升漏洞。成功利用此漏洞的攻击者可以对系统执行打开文件的用户可以执行用户可以在目标用户的上下文中在网站上执行的任何操作，包括执行任意命令。在电子邮件攻击情形中，攻击者可以通过向用户发送特制的链接并诱使用户单击该链接来利用此漏洞。攻击者无法强迫用户访问网站。相反，攻击者必须说服用户采取行动，通常是单击电子邮件或即时消息中的链接。

要在“常见漏洞和披露”列表中以标准条目查看此漏洞，请参阅 [CVE-2011-3417](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-3417)。

#### ASP.NET 表单身份验证票证缓存漏洞 (CVE-2011-3417) 的缓解因素

缓解是指一种设置、通用配置或者一般的最佳实践，它以默认状态存在，能够降低利用漏洞的严重性。以下缓解因素在您遇到的情形中可能会有所帮助：

-   默认情况下，ASP.NET 响应不被 OutputCache 缓存。站点的开发人员必须通过页面上的 OutputCache 指令选择输出缓存。
-   成功利用此漏洞的攻击者可以获得与目标用户相同的用户权限。那些帐户被配置为拥有较少系统用户权限的用户比具有管理用户权限的用户受到的影响要小。
-   默认情况下，IIS 不会安装在任何受影响的操作系统版本上。仅手动安装此组件的客户可能容易受到此问题的影响。
-   默认情况下，安装 .NET 而不安装 ASP.NET。只有手动安装和启用 ASP.NET 的客户才可能受此问题的影响。

#### ASP.NET 表单身份验证票证缓存漏洞 (CVE-2011-3417) 的变通方法

变通办法是指一种设置或配置更改，它不能从根本上纠正漏洞，但有助于在应用更新之前封堵已知的攻击源。Microsoft 已测试了以下变通方法，并在讨论中指明了变通方法是否会降低功能性：

-   **限制表单验证 Cookie 为 SSL 协议**

    在进行备份后打开此应用程序或全局 web.config 文件

    对 &lt;forms&gt; 元素设置 requireSSL="true"，如以下代码中所示。

    ```
    <forms loginUrl="Secure\Login.aspx"
    requireSSL="true" />
    ```

    **如何撤消变通方法。**

    从备份副本恢复 web.config 文件。

-   **为表单身份验证 Cookie 禁用可调到过期**

    1.  在进行备份后打开此应用程序或全局的 web.config 文件
    2.  对 &lt;forms&gt; 元素设置 slidingExpiration="false"，如以下代码中所示。

        `<forms slidingExpiration="false"/>`

    **变通办法的影响。** 用户将被迫在他们的票证到期后重新登录。

    **如何撤消变通方法。**

    从备份副本恢复 web.config 文件。

-   **禁用 OutputCache**

    1.  备份 Web 应用程序文件夹。
    2.  删除网页中的所有 &lt;%@ OutputCache %&gt; 指令。
    3.  如果您的网站是 MVC 应用程序，还请删除控制器操作中的所有 OutputCache 属性。

    **变通办法的影响。** 用户将被迫在他们的票证到期后重新登录。

    **如何撤消变通方法。**

    恢复 Web 应用程序文件夹。

#### 部署信息

**安装更新**

当您安装此安全更新时，安装程序将检查系统中正在更新的一个或多个文件是否已由 Microsoft 修补程序更新。

如果以前安装的修补程序已经更新过其中的某个文件，安装程序就会将 RTMQFE、SP1QFE 或 SP2QFE 文件复制到系统中。否则，安装程序将 RTMGDR、SP1GDR 或 SP2GDR 文件复制到您的系统中。安全更新可能不包含这些文件的所有版本。有关此行为的详细信息，请参阅 [Microsoft 知识库文章 824994](http://support.microsoft.com/kb/824994)。

有关安装程序的详细信息，请参阅 [Microsoft 知识库文章 832475](http://support.microsoft.com/kb/832475)。

有关此公告中出现的术语详细信息（如*修补程序*），请参阅 [Microsoft 知识库文章 824684](http://support.microsoft.com/kb/824684)。

    2.  当文件显示在“**程序**”之下时，右键单击文件名，然后单击“**属性**”。
    3.  在“**常规**”选项卡上，将文件大小与公告知识库文章中提供的文件信息表进行比较。
        **注意** 根据系统上安装的操作系统或程序的版本，文件信息表中所列的某些文件可能并未安装。
    4.  您也可以单击“**详细信息**”选项卡，并将信息（如文件版本和修改日期）与公告知识库文章中提供的文件信息表进行比较。

    5.  最后，您也可以单击“**以前的版本**”选项卡，并将文件的以前版本的文件信息与新的或经过更新的文件版本的文件信息进行比较。
